# Gemini 測試案例生成 SOP (標準作業流程)

這份文件記錄了與 Gemini CLI 互動時，關於測試案例生成的所有約定與偏好設定。它旨在確保每次產出的測試案例都能符合特定的格式、內容與 logique。

## 1. 檔案產出流程
當您交付一個任務時，Gemini 會遵循以下流程產出檔案：
1.  根據任務來源（例如 `Superdsp phase 1.7.0`），在 `generated_test_cases` 資料夾內建立一個同名的子資料夾。
2.  遵循檔案命名規則產生檔案。
3.  將產出的所有檔案放入新建的對應子資料夾中。

## 2. 檔案命名規則
所有產出的檔案，檔名都需要加上時間戳記，格式為 `_YYYYMMDD_HHMMSS`。

*   **測試案例檔案**：
    `[來源資料夾名稱]_test_case_[時間戳記].csv`
    (例如：`Superdsp_phase_1.7.0_test_case_20250915_150000.csv`)

*   **User Story 分析檔案**：
    `[來源資料夾名稱]_UserStory_[時間戳記].csv`
    (例如：`Superdsp_phase_1.7.0_UserStory_20250915_150100.csv`)

*   **跨多個資料夾的分析**：
    若任務橫跨多個資料夾，則使用 `System_Analysis` 作為前綴。
    (例如：`System_Analysis_UserStory_20250915_150100.csv`)

## 3. 檔案格式
*   所有產出檔案均為 `.csv` (逗號分隔值) 格式。
*   無法直接產出多工作表 (multi-sheet) 的 Excel (`.xlsx`) 檔案。

## 4. 測試案例產出風格
Gemini 具備兩種測試案例產出風格，預設為「樣式Ａ」。

### 樣式Ａ (預設風格)
這是我與您共同建立的、最完整且詳細的測試案例產出風格。它旨在從 User Story 出發，全面覆蓋測試情境。

**核心規則：**
*   **優先從 UserStory 發想案例**：即使文件中沒有明確的「UserStory」章節，也會從「功能細項」、「驗收標準」、「流程圖」等部分反向推導使用者情境。
*   **測試案例分解**：每個 User Story 都會被充分分解為一個或多個測試案例，涵蓋所有相關類型。

**欄位與內容：**
| 欄位名稱 | 說明 | 範例內容 |
|---|---|---|
| `系統` | 測試案例所屬的系統 (e.g., SuperDSP, OYM, ERP/OSS) | `SuperDSP` |
| `權限` | 執行測試所需的使用者權限 | `Media Admin` |
| `功能 / 頁面` | 測試案例所針對的具體功能或頁面 (不含系統名稱前綴) | `RTB 底價設定` |
| `測試功能` | 測試的目的或行為 (使用 `【】` 標籤分類) | `【正向】更新底價` |
| `測試情境` | 描述測試的具體場景或使用者意圖 | `媒體管理者成功設定並更新一個廣告格式的底價` |
| `操作步驟` | 執行測試的詳細步驟 | `1. 登入 OYM 平台。<br>2. 進入「RTB 底價設定」頁面。` |
| `期望結果` | 測試預期的結果 (條列化) | `1. 系統提示「儲存成功」。<br>2. 介面上底價數字立即更新。` |
| `備註` | 應詳實記錄該案例參考的**所有**相關文件，並盡量標示出所參考的具體**章節標題**，以達到最佳可追溯性。 | `主文件：ITO-[Commerce AD Phase 3]...pdf<br> - 章節：UserStory 與系統行為<br>細節文件：ITO-[功能細項] ERP...pdf<br> - 章節：新增 RTB 合約` |

**格式化細則 (2025/09/26 更新):**
*   **權限欄位命名**: 權限應使用具體角色，而非模糊的描述。
    *   **外部使用者**: `Media User`, `Media Admin`, `Agency User`, `Agency Admin`。
    *   **內部使用者**: `Onead User`。若規格文件能識別出更詳細的團隊/角色，應加上括號，例如：`Onead User (PAD)`, `Onead User (PM)`。
*   **內容換行格式**: 在 `操作步驟` 與 `期望結果` 等欄位中，當產出 CSV 檔案時，條列式項目應使用**實際換行** (newline character) 來分隔，而不是使用 HTML 的 `<br>` 標籤。
*   **結尾符號**: 在 `測試情境`、`操作步驟`、`期望結果` 的任何一行結尾，都**不需**加上句號 `。`。
*   **測試功能標籤排序**: 「測試功能」欄位中的標籤應遵循以下固定順序排列：`[權限]`, `[功能新增]`, `[正向]`, `[反向]`, `[整合]`, `[情境]`, `[ＵＩ]`, `[UI/UX]`。
*   **備注來源追溯**: 當依據一個既有的測試案例檔案來產出新的測試案例時，新案例的「備注」欄位不應再將來源標示為舊的測試案例檔。應回溯並填入該案例最原始的規格文件（例如 PDF）作為來源，並附上對應的章節標題。

**內容排序：**
測試案例的內容會依照以下順序組織：
1.  **權限**相關驗證
2.  **合約**相關驗證
3.  **一般功能**及欄位驗證
4.  **數據報表**及帳務管理

### 樣式Ｂ
這是我比較早期、較簡潔的測試案例產出風格。當您需要快速、直接的功能驗證時可使用。

**特點：**
*   欄位較少 (例如不包含「系統」、「測試情境」、「備註」)。
*   內容直接根據功能規格發想，風格較為精簡，沒有特定的內容排序或分類標籤。

## 5. 系統別名
*   系統 `ERP` 在規格文件中也可能被稱作 `OSS`。

## 6. 記憶與可攜性
*   我的記憶與您的**使用者帳號**綁定，而非專案檔案。
*   在不同電腦上使用**同一個帳號**，記憶會自動同步。
*   若使用**不同帳號**，需要透過「初始化指令」來載入 SOP。
