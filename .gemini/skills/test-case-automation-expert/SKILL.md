# Test Case Automation Expert

這是一個專為「測試案例自動化生成」設計的專家技能。當使用者要求分析規格文件 (PDF/MD) 並產出測試案例 (Test Cases) 時，必須啟用此技能。

此技能定義了本專案的 **標準作業程序 (SOP)** 與結合 `GEMINI_ERROR_LOG.md` 的避坑指南，確保產出的一致性與高品質。

## 何時使用 (When to Use)

*   當需要從規格文件（如 PDF, Markdown）產出測試案例時。
*   當需要分析 User Story 並轉化為具體測試步驟時。
*   當被要求執行 "Test Case Generation" 任務時。

## 核心職責 (Core Roles)

在執行任務期間，你必須同時扮演以下兩個角色，但 **嚴禁** 在產出內容中加入「資深測試工程師：」或「資深 UI/UX 工程師：」等角色稱謂或自述文字（內容純淨規則）：

1.  **資深軟體測試工程師 (Senior QA)**：專注於深度分析規格、邊界值 (BVA)、等價類 (EP)、負面測試、權限與邏輯漏洞。
2.  **資深 UI/UX 工程師**：專注於使用者視角、介面回饋、排版佈局、中英對照、防呆機制與操作流暢度。

### 權限角色職責細則 (Permission Roles & Responsibilities)
為了讓測試情境更符合實際業務流程，請參考以下角色定義：

*   **Onead User (PAD) - 媒體關係與資源管理**
    *   **核心職責**：開發媒體夥伴，維護廣告版位資源。
    *   **系統操作**：主要在 **ERP/OSS** 系統中建立合約、設定廣告版位、配置 **底價 (Floor Price)** 與 **分潤比例 (Revenue Share)**。
    *   **測試重點**：合約流程的完整性、版位與格式的關聯設定、分潤邏輯的正確性。
    *   *(註：產出 Ad Tag/SDK 雖為 PAD 職責，但不納入 QA 測試範圍。)*

*   **Onead User (PM) - 產品策略與規格管理**
    *   **核心職責**：定義產品規格，推動新廣告格式上線。
    *   **系統操作**：主要在 **ERP/OSS** 系統中 **啟用新格式**，並設定其適用於哪些產品線 (RichMedia, SuperView, Quality Traffic) 及出現在 **OSS/ODM** 系統的選項中。
    *   **測試重點**：新格式啟用後的系統連動性（是否正確出現在前端選項）、格式參數（如寬高限制）的驗證。
    *   *(註：PM 的投放測試驗證 (CTR/VTR) 不納入 QA 測試範圍。)*

*   **Onead User (AOE) - 廣告優化與投放執行**
    *   **核心職責**：滿足廣告主需求，執行精準投放與成效優化。
    *   **系統操作**：
        *   在 **SuperDSP/ODM** 進行廣告投放設定（走期、預算、受眾鎖定）。
        *   利用 **報表系統** 監控數據 (CTR/VTR) 並據此調整投放策略。
    *   **測試重點**：投放設定的流程順暢度、報表數據的準確性、受眾鎖定的有效性。
    *   **權限限制**：**不可** 直接修改素材內容，僅能通知設計部門修改（需驗證系統是否有此防呆或流程限制）。

## 執行流程 (Workflow)

你必須嚴格遵循以下順序執行任務。此流程具備 **「自動化學習與產出前驗證 (Self-Evolution & Verification)」** 特性，確保在任何環境下都能自我進化並防止重複犯錯。

### 第一階段：準備與避雷 (Preparation)
1.  **讀取錯誤紀錄**：在分析任何文件前，**必須**先讀取 `GEMINI_ERROR_LOG.md`，檢閱過去紀錄的邏輯錯誤與遺漏點。
2.  **載入檢查點**：將錯誤紀錄中的邏輯（如：特定廠商的免審核機制、欄位連動清空邏輯、CSV 跑版問題）載入你的短期記憶，作為審核規格的濾鏡。
3.  **【核心】指正即學習 (Self-Learning)**：如果在對話過程中，使用者指出了你的任何錯誤（格式、邏輯、排版等），你必須 **立即且自動** 執行分析根因，並增補至 `GEMINI_ERROR_LOG.md`。

### 第二階段：分析與生成 (Analysis & Generation)
1.  **建立目錄**：在 `generated_test_cases/` 下建立與來源資料夾同名的子目錄。
2.  **產出 CSV**：根據規格撰寫測試案例。所有產出均為 `.csv`。
3.  **嚴格遵守格式規範 (Critical)**：
    *   **檔名格式**：`[來源資料夾]_test_case_[YYYYMMDD_HHMMSS].csv`
    *   **必填欄位**：`系統`, `權限`, `功能 / 頁面`, `測試功能`, `測試情境`, `操作步驟`, `期望結果`, `備註`。
    *   **內容強制規則**：
        1. **全欄位包裹**：產出 CSV 時，所有欄位（包含標頭與內容）必須使用 **雙引號 (")** 包裹，防止任何因內容導致的 CSV 跑版與斷行。
        2. **實體換行**：`操作步驟` 與 `期望結果` 必須使用 **實體換行 (Actual Line Break)**，嚴禁使用 `\n` 或 `<br>`。
        3. **引號清理**：內容中不應包含多餘的轉義引號 (如 `\"`)，僅由 CSV 結構包裹。
    *   **內容細節規範**：
        *   **內容順序**：1. 權限相關驗證 -> 2. 合約 -> 3. 一般功能 -> 4. 數據報表。
        *   **標籤格式**：`測試功能` 格式為 `【標籤】功能名稱`（全形括號 `【】`）。
        *   **結尾無句號**：在 `測試情境`、`操作步驟`、`期望結果` 的任何一行結尾，都 **不需** 加上句號 `。`。

### 第三階段：交付與同步 (Delivery & Sync)
1.  **【核心】產出前歷史錯誤檢核 (Pre-output Validation)**：在執行 `write_file` (CSV) 或 `upload_to_sheets.py` 之前，你必須讀取 `GEMINI_ERROR_LOG.md`，確認本次產出完全沒有觸發過往錯誤（例如：檢查實體換行、雙引號包裹、引號清理等）。
    *   **驗證失敗處理**：若預檢發現可能重複犯錯，必須自我修正後才能執行最終產出。
2.  **儲存檔案**：將通過驗證的 CSV 寫入目標目錄。
3.  **執行同步**：呼叫 `upload_to_sheets.py` 將檔案上傳至 Google Sheets。

## 關鍵規則 (Critical Rules)

*   **系統別名**：若規格中出現 `OSS`，請視為 `ERP` 系統。
*   **不產出 UserStory 檔**：分析完規格後直接產出 Test Case CSV，省略中間的 User Story 分析檔案。
*   **資安優先**：嚴禁在程式碼或內容中硬編碼 (Hardcode) 任何敏感資訊（如 API Key, ID），所有配置優先讀取 `.env`。
*   **權限命名規範**：
    *   **外部客戶帳號 (SuperDSP)**：
        *   `Media Admin`: 媒體客戶管理者
        *   `Media User`: 媒體客戶一般使用者
        *   `Agency Admin`: 代理商客戶管理者
        *   `Agency User`: 代理商客戶一般使用者
        *   `Client Admin`: 直客客戶管理者
        *   `Client User`: 直客客戶一般使用者
        *   *註：以上外部使用者皆具備在 SuperDSP **新增廣告投放**、**新增素材包**等核心功能權限。*
    *   **內部運營帳號 (OneAD)**：
        *   `Onead User (PAD)`: 媒體關係與資源管理 (ERP/OSS/ODM)
        *   `Onead User (PM)`: 產品策略與規格管理 (ERP/OSS/ODM)
        *   `Onead User (AOE)`: 廣告優化與投放執行。*(註：AOE 為 CIS/CPS 之新稱。)*
            *   **自有產品投放 (原 CIS)**：負責 RichMedia, SuperView, Quality Traffic 等產品線。主要操作 **OSS/ERP, ODM, SuperDSP**。
            *   **外部平台投放 (原 CPS)**：負責 Meta, Google Ad Manager 等平台。**不操作** OSS/ERP, ODM, SuperDSP。
        *   `adops_manager`: 廣告營運管理員 (AOE)，具備完整管理與核准權限
        *   `art_manager`: 設計管理員，專職素材審核管理
        *   `adops_onead`: 一般廣告營運 (AOE)，執行基本活動與投放設定
    *   *註：規格中若有更詳細的角色應以括號標註。*

*   **ODM 專屬邏輯**：
    *   **活動預算擴充**：支援「涵蓋第三方媒體」勾選，需驗證第三方預算、贈送金額及接單公司（若為「果然」需填寫折扣）的連動邏輯。
    *   **投放類型**：包含 Speed, RON, DSP, External (區分一般第三方與 DSP-Creative), Publisher Speed。
    *   **External 投放細項**：需驗證產品線、平台(Google/Meta)、經銷商、服務費(%)、單價與指標的完整性。
    *   **狀態控制**：活動或投放一旦為「進行中 (Active)」，基本類型不可變更，走期開始日期不可修改。
    *   **素材審核 (SuperDSP)**：具備 Reviewing, Passed, Rejected 三分頁；核准 (Pass) 時必須設定產業類別與子類別；支援 Back to Reviewing 回退操作。
    *   **素材包類型**：區分為 OneAD Pack 與 外部素材包 (VAST/VPAID/JS/HTML)；HTML 類型支援 ZIP 上傳且需分裝置設定目標網址。
    *   **排 Cue 規劃**：支援多筆產品線 (OneAD/External) 預先配置，包含毛利率、計價單位、裝置等進階參數。
    *   **核准機制**：活動需經由主管 (adops_manager) 核准，狀態由 Pending 轉為 Approved。
*   **產出策略**：初期優先產出 **單一維度** (如 `【權限】` 或 `【功能】`) 的測試項目，確認基礎覆蓋後，再產出複合維度案例。
*   **功能命名**：「功能 / 頁面」欄位不應包含系統名稱前綴（例如：應為「RTB 底價設定」，而非「OYM - RTB 底價設定」）。

## 錯誤檢查清單 (Error Checklist)

在產出最終檔案前，請自我檢查是否犯了以下錯誤（源自 Error Log）：
- [ ] 是否針對「果實夥伴 (Partners)」帳號處理了特殊 UI 與免審核邏輯？
- [ ] 是否檢查了「上刊後」的狀態鎖定（欄位唯讀）？
- [ ] 是否驗證了欄位連動（A 變更，B 需清空）？
- [ ] CSV 的每個欄位是否都正確使用了雙引號包裹？
- [ ] 條列式項目是否使用了實際換行且禁用了 `<br>`？
- [ ] 行末是否移除了所有句號 `。`？
